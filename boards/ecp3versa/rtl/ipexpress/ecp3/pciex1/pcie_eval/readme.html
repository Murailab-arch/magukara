PCI Express IP Core

Implementing the IP Module Using ispLEVER(Lattice Diamond SW)

Configuration Options for Simulation

The evaluation testbench supplied with the core requires that the following options are NOT selected to ensure proper functional simulation:

1.      Do not select: Include Wishbone Interface

2.      Do not select: Include ECRC support

All other options are user selectable.


Instantiating the Core

The generated PCI Express IP core package includes black-box (<user_name>_bb.v) and instance (<username>_inst.v) templates that can be used to instantiate the core in a top-level design.

An example RTL top-level reference source file that can be used as an instantiation template for the IP core is provided in \<project_dir>\pcie_eval\<username>\src\top. Users may also use this top-level reference as the starting template for the top-level for their complete design.

For ispLEVER Linux/Unix version*,the user should manually add an environment variable named "SYNPLIFY" to indicate installation path of Synplify tool in the local environment file. For example:
setenv SYNPLIFY /public/install/isplever8_1_linux/isptools/synplify_linux.

* Note: No need to set this for Lattice Diamond SW.

 

Hardware Evaluation

Lattice's IP hardware evaluation capability makes it possible to create IP cores that operate in hardware for a limited period of time (approximately four hours) without requiring the purchase on an IP license. The hardware evaluation capability is enabled by default. It can be disabled by right clicking on "Build Database" in the "Process for current sources" window of the Project Navigator. The setting is called "Hardware Evaluation" and the options are "Enable" or "Disable".
 
When the Hardware Evaluation feature is enabled in the design, it will generate a programming file that may be downloaded into the device. After initialization, the IP core will be operational for approximately four hours. After four hours, the device will stop working and it will be necessary to reprogram the device to re-enable operation. This hardware evaluation capability is only enabled if the core has not been licensed. During implementation, a license check is performed. If the hardware evaluation feature is disabled, a pop-up window will be displayed indicating a license failure. Click"OK" in the window and the bitstream will not be generated. If a license is detected, no pop-up window is displayed and core generation is completed with no restrictions.

 

Implementing the Core in a Top-Level Design


As described previously, the top-level file <username>_top.v/vhd provided in \<project_dir>\pcie_eval\<username>\src\top supports the ability to implement just the PCI Express Block.

Push-button top-level implementation of this top-level is supported via the ispLEVER project file <username>_eval.syn(.ldf for Lattice Diamond SW tool) located in \<project_dir>\pcie_eval\<username>\impl\<synplify/precision>.

This design is intended only to provide an accurate indication of the device utilization associated with the core itself and should not be used as an actual implementation example.


To use the project file:

  • Select Open Project under the File tab in ispLEVER(File->Open->Project in Lattice Diamond).
  • Browse to \<project_dir>\pcie_eval\<username>\impl\<synplify/precision> in the Open Project dialog box.
  • Select and open <username>_eval.syn(.ldf for Lattice Diamond SW tool). At this point, all of the files needed to support top-level synthesis and implementation will be imported to the project.
  • One lpf file is available in the \impl\<synplify/precision> directory <username>_eval.lpf.
  • All other options remain at their default values.
  • Implement the complete design via the standard ispLEVER(Lattice Diamond) GUI flow.

Note: The device selected in IPexpress must match the device selected in ispLEVER(Lattice Diamond).  If the device is changed in ispLEVER(Lattice Diamond), a new IPexpress implementation must be created.

Note: The SCM15 device and the SCM25 device in a 900-pin package are not supported by the core only design as the number of external I/O pins exceeds the number of available external I/O pins.

 

For ispLEVER tool Linux/Unix version*, the top-level synthesis needs to be run separately with Synplify Pro as the ispLever Linux/Unix version only accepts an EDIF entry. Synthesis tcl files will be generated for this purpose including pci_exp_top.tcl in the \<project_dir>\pcie_eval\<username>\impl\synplify directory

In order to generate edif on the Linux/Unix platform from a GUI interface, the user should do the following:

  • Type "$SYNPLIFY/bin/synpwrap -gui" for OEM synplify pro or "$SYNPLIFY/bin/synpwrap-gui -notPro" for OEM synplify.
  • Select "Run" and "Run Tcl scripts".
  • Browse to \<project_dir>\pcie_eval\<username>\impl\synplify.
  • Choose pci_exp_top.tcl and click Open.
  • All other options remain at their default values.
  • Implement the complete design via the standard ispLEVER GUI flow.

In order to generate edif on the Linux/Unix platform from a command line, the user should do the following:

  • Change to the \<project_dir>\pcie_eval\<username>\impl\synplify directory.
  • Type "synplify_pro -batch pci_exp_top.tcl .
  • Browse to \<project_dir>\pcie_eval\<username>\impl\synplify.

To use the project file:

  • Select Open Project under the File tab in ispLEVER.
  • Browse to \<project_dir>\pci_exp_eval\<username>\impl\synplify in the Open Project dialog box.
  • Select and open <username>_eval.syn. At this point, all of the files needed to support top-level synthesis and implementation will be imported to the project.
  • Implement the complete design via the standard ispLEVER GUI flow.

* Note: This step is not necessary for Diamond Linux,implementing flow is same as the Windows platform as described above.

 

Running Functional Simulation

The functional simulation includes a configuration-specific behavioral model of the PCI Express, which is instantiated in an FPGA top level along with some test logic (PLLs, UMI and Sysbus). This FPGA top is instantiated in an eval testbench that configures FPGA test logic registers and PCI Express IP core registers. The testbench files can be found in \<project_dir>\pcie_eval\<username>\testbench.


The generated IP core package includes the configuration-specific behavior model (<username>_beh.v) for functional simulation in \<project_dir>. ModelSim SE simulation is supported via testbench files provided in \<project_dir>\pcie_eval\<username>\testbench. Models required for simulation are provided in the corresponding \<project_dir>\models folder.

Users may run the evaluation simulation on ModelSim SE (Windows or Linux) by doing the following:

  • Open ModelSim SE.
  • Under the File tab, select Change Directory
  • Set the directory to /<project_dir>/pcie_eval/<username>/sim/modelsim/rtl
  • Select OK.
  • Under the Tools tab, select TCL, then select Execute Macro.
  • Select file ../script/eval_beh_rtl_se.tcl for modelsim SE;
  • Select Ok.

 

Users may run the eval simulation on Active-HDL(Windows only) by doing the following:

  • Open Active_HDL.
  • Under the Tools tab, select Execute Macro...
  • Select file /<project_dir>/pcie_eval/<username>/sim/aldec/scripts/eval_beh_rtl.do
  • Select OK.


NOTE1: When the simulation completes, a pop-up window will appear asking "are you sure you want to finish?" Answer "no" to analyze the results (answering "yes" closes ModelSim SE).

NOTE2: Simulation will only be available for the default configuration.

NOTE3: VHDL Simulationusing ModelSim SE is not supported in the evaluation package.

Special instructions for using ModelSim libraries

The ispLEVER(Lattice Diamond) CD does not include library elements for black box elements in the directory <isplever(diamond)_dir>\cae_library\simulation\blackbox. A patch is available on the Lattice Semiconductor web site that provides the missing black box elements.

 

Once the missing black box library elements are loaded, the appropriate zip files can be unzipped in the blackbox directory. For this IP core, the following zip files will need to be unzipped in order to compile with ModelSim SE:

 

pcsd-mti_6.0-V1-1.zip

pcsc-mti_6.0-V1-1.zip

pcsa-mti_6.0-V2-1.zip

 

Unzip them under<isplever(diamond)_dir>\cae_library\simulation\blackbox, so you will see subdirectories as pcsa_work,pcsc_work,pcsd_work.

ispLever 7.2 SP1 Known issues

To validate the IP core in ECP3 device on board with ispLever 7.2SP1, users need to do the following:

  • Implement the complete design via the standard ispLEVER GUI flow.
  • Browse to \<project_dir>\pcie_eval\<username>\impl\synplify.
  • Remove the following line in pcs_pipe_8b_xn.txt
    • CH0_TDRV                "1"
    • CH1_TDRV                "1"
    • CH2_TDRV                "1"
    • CH3_TDRV                "1"
  • Add the following lines in pcs_pipe_8b_xn.txt
    • CH0_TDRV_AMP      "2"                
    • CH1_TDRV_AMP      "2"                
    • CH2_TDRV_AMP      "2"
    • CH3_TDRV_AMP      "2"
  • Run the process “Generate Bitstream Data” to generate the “pcs_pipe_8b_xn.ptx”
  • Add following lines to file “pcs_pipe_8b_xn.ptx”
    • ch0_TDRV_DRVCUR_SET      "100"
    • ch1_TDRV_DRVCUR_SET      "100"
    • ch2_TDRV_DRVCUR_SET      "100"
    • ch3_TDRV_DRVCUR_SET      "100"
    • ch0_dconuoflsb        "000"
    • ch1_dconuoflsb        "000"
    • ch2_dconuoflsb        "000"
    • ch3_dconuoflsb        "000"
    • ch0_dcostep  "11"
    • ch1_dcostep  "11"
    • ch2_dcostep  "11"
    • ch3_dcostep  "11"
    • ch0_rsvd10    "100"
    • ch1_rsvd10    "100"
    • ch2_rsvd10    "100"
    • ch3_rsvd10    "100"
  • Run the process “Generate Bitstream Data”
